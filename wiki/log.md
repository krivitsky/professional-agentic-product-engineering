## [2026-07-04] restructure | Granular tree — one page per tip (guide → tier → tips)

Rebuilt the vault from a flat 22-page view into a **3-level tree**. The old graph was a star: every page linked the root `guide.md`, so all content orbited one `guide` node and the tier structure was invisible.

- **Renamed** the 8 tier pages to the guide's em-dash heading form: `Tier N — <Name>` (colon avoided — unsafe in Obsidian/macOS filenames). Rewired every inbound `[[wikilink]]`.
- **Created 60 per-tip pages** (`concepts/Tip N.M — <Title>.md`) — one per guide tip, with the verbatim Instead/Prefer blockquote, a one-line gloss, and `## Related` links to the tier + prev/next sibling (+ occasional verified concept cross-link). T2–T8 generated by 7 parallel subagents (one per tier); T1 hand-built as the template.
- **Tree rule:** tip pages attribute the guide as *plain text* `` `guide.md #tip-N-M` `` and link **only their tier** (`**Part of:** [[Tier N — …]]`) — no `[guide.md](../../guide.md)` markdown link. Only tier/concept/entity pages link the root guide. Result: tips cluster under their tier; guide connects to tiers, not tips.
- **Filename edge cases:** slashes are illegal in filenames — `Tip 5.5` uses a division slash `∕` (U+2215); `Tip 3.3`'s `/clear` uses a slash-free filename with `/clear` kept in the H1 for command fidelity. Filenames = inbound wikilinks in every case.
- **Schema updated** (`CLAUDE.md`): added "Vault structure — the guide as a 3-level tree" codifying the tier-hub + per-tip page format, the plain-text-attribution/no-guide-link rule, and the granular re-wikify procedure so future `guide.md` edits land as tip pages, not a flat vault. Root repo `CLAUDE.md` sync rule #1 updated to match.
